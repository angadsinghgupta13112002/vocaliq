Google Cloud Firestore — stores all user data and
coaching session results.

Google Cloud Storage — stores the uploaded video and
audio files.

Google Cloud Run — where the entire application is
deployed as a containerized service with 2 vCPUs and
2 gigabytes of RAM to handle large video processing.

Google Analytics GA4 — tracks user behavior on both
the frontend and backend.

And Google Cloud Logging — captures every server
event through Cloud Run's built-in log collection.

All of these are fully working and live right now.

SLIDE 7 — Google Cloud Vision + MediaPipe Gesture Detection
─────────────────────────────────────────────────────────

Thank you Abhinay. I'm Samuel Paul Chetty, and I'll
walk you through the two visual AI systems powering
VocalIQ — Cloud Vision for emotions and MediaPipe for
gestures — then cover Gemini, Analytics, Logging,
and our project summary.

Let me start with the emotion analysis pipeline.

EMOTION TIMELINE — CLOUD VISION

Google Cloud Vision is what builds the second-by-second
emotion timeline you saw in the report.

[OPEN: VS Code → client/src/utils/frameExtractor.js]

For locally recorded and uploaded videos, step one
happens entirely in the browser. Our frameExtractor.js
uses the HTML5 Video element and Canvas API to extract
one JPEG frame every three seconds — up to twenty frames.
Each frame is captured at 320 by 240 resolution and
encoded as a base64 JPEG string. No server roundtrip,
no extra dependencies.

[OPEN: VS Code → server/utils/serverFrameExtractor.js]

For Google Drive videos — which the client never
downloads — we run frame extraction server-side.
Our serverFrameExtractor.js writes the video buffer
to a temporary file, calls ffmpeg to extract JPEG
frames at the same three-second interval, reads them
back as base64 strings, then cleans up all temp files.
This runs in parallel with the Gemini analysis so
it adds near-zero extra wall-clock time.

[OPEN: VS Code → server/services/visionService.js]

Step two is identical regardless of where the frames
came from. Our visionService.js sends those frames to
the Google Cloud Vision Face Detection API in batches
of five — to stay within rate limits — using Promise.all.

For each frame, Cloud Vision returns likelihood scores
for four emotions — joy, sorrow, anger, and surprise.
We map these to coaching labels. High joy means
confident. High sorrow means nervous. Combined sorrow
and low joy means anxious.

Cloud Vision also returns pan and tilt angles for
the face. If the speaker is turned more than 15 degrees
horizontally or 20 degrees vertically, we flag that
frame as looking away — this is how we build the eye
contact percentage you see in the report.

[SHOW: the Emotion Timeline tab on the coaching report]

The result is the emotion timeline — each second mapped
to an emotion, eye contact direction, and confidence
score. The summary card shows dominant emotion, nervous
moments, confident moments, and eye contact percentage.

GESTURE DETECTION — MEDIAPIPE

[OPEN: VS Code → client/src/utils/gestureExtractor.js]

For locally recorded and uploaded videos, gesture
detection runs entirely in the browser using the
MediaPipe HandLandmarker model — loaded once from
a CDN and cached as a singleton.

The extractor seeks through the video at three-second
intervals, draws each frame onto a canvas, and passes
it to MediaPipe for landmark detection. MediaPipe returns
21 landmark coordinates per hand — from wrist to each
fingertip.

We classify gestures from those landmarks using a simple
rule system — we check whether each finger is extended
by comparing the y-coordinate of the fingertip to the
metacarpal base. A finger is extended if the tip is
meaningfully above the base joint. Combined with thumb
direction detection and priority ordering, we classify
eight gesture types: open hand, pointing, peace,
thumbs up, fist, partial open, neutral hand, and
no hands.

[SHOW: the Gestures tab on the coaching report]

The result is the gesture timeline — a color bar over
time, an expressiveness score combining hands-visible
percentage and expressive gesture rate, a breakdown
chart by gesture type, and a coaching tip specific to
the dominant gesture.

For Google Drive videos where the client can't run
MediaPipe, Gemini handles gesture analysis in a
dedicated third pass — which Samuel will cover
in the next slide.

SLIDE 8 — Google Gemini (Three-Pass Pipeline)
─────────────────────────────────────────────────────────

The Gemini API is the core of VocalIQ. We use
Gemini 2.5 Flash through a three-pass pipeline —
two passes for speech coaching and a third for
gesture analysis on Drive-uploaded videos.

Let me explain the design.

[OPEN: VS Code → server/services/geminiCoachingService.js]

A naive approach sends the video to Gemini with one
prompt asking for everything at once. The problem is
that Gemini produces generic, surface-level advice
when asked too many things simultaneously.

We wanted coaching grounded in what the speaker
actually did — specific to their recording.

[SCROLL to the analyzeVideoWithGemini function]

In Pass 1, we write the video buffer to a temporary
file and upload it to the Gemini File API — a dedicated
endpoint for large media files. We poll until the file
is in ACTIVE state, meaning Gemini has fully processed
the video and it's ready for multimodal queries.

We then send a structured prompt asking Gemini to act
as a speech assessment engine and return JSON.

That JSON contains the overall score, five sub-scores
across content clarity, emotional delivery, vocal
confidence, pacing, and audience engagement, the
emotion detected versus the emotion needed for this
scenario, the speaker's words per minute, and a full
timestamped transcript broken into segments with issue
tags and suggested fixes.

[POINT to Pass 1 section]

Everything on the Overview and Transcript tabs
came from Pass 1.

[SCROLL to Pass 2 section]

Pass 2 takes the entire Pass 1 JSON output and injects
it as context into a second prompt. We tell Gemini —
here is what you observed. Now act as a personal
speaking coach and build a detailed improvement plan
based specifically on these findings.

Because Pass 2 has Pass 1's analysis as its starting
point, the coaching is grounded in what was actually
observed. It cannot hallucinate generic advice.
It returns detailed tip cards, vocal control data,
strengths to keep, and a seven-day practice plan.

[SCROLL to Pass 3 section]

Pass 3 is the gesture analysis pass — it only runs
for Google Drive videos where MediaPipe cannot run
client-side, since the video never reaches the browser.

Because the video file is already uploaded and ACTIVE
in the Gemini File API from Pass 1, Pass 3 re-uses the
same file URI. There is zero extra upload cost.

We send a prompt asking Gemini to analyze hand gestures
at three-second intervals and return a JSON array where
each entry has a second, a gesture label, and a hand
count. The gesture labels match exactly what MediaPipe
uses for local recordings — so the report looks
identical regardless of how the video was submitted.

[POINT to the gesture prompt text in the file]

After Pass 3, we delete the uploaded file from the
Gemini File API and return all three results to the
controller.

The key insight across all three passes: each pass
uses what came before. Pass 2 is better because it
has Pass 1. Pass 3 re-uses the already-processed file.
Every decision in this pipeline was made to maximize
quality without paying unnecessary latency costs.

SLIDE 9 — Google Analytics
─────────────────────────────────────────────────────────

VocalIQ uses Google Analytics 4 at two separate levels
— the frontend and the backend — and I want to explain
both because they work differently.

On the frontend, we load the gtag.js library in
index.html. We also built a custom analytics.js module
with trackPageView and trackEvent functions.

A PageViewTracker component inside React Router calls
trackPageView on every navigation — so every page in
the single-page app is tracked as a separate page view.

We fire custom events at every meaningful user action.
Login events when OAuth completes. session_submitted
when a user clicks Analyze. session_complete when
analysis finishes. report_tab_viewed with the tab
name every time a user switches tabs on the report —
including the new Emotion Timeline and Gestures tabs.

[OPEN: analytics.google.com → Reports → Realtime]

This is the GA4 Realtime dashboard. It shows active
users right now and which events are firing.

[OPEN: Reports → Engagement → Events]

The Events report shows every custom event — how many
times it fired and how many unique users triggered it.
session_submitted and session_complete together give
us our pipeline completion rate. report_tab_viewed
tells us which parts of the report users actually read.

Now for the backend analytics. We built an analytics
service on the Express server that uses the GA4
Measurement Protocol — an HTTP API that lets the
server send events directly to Google Analytics
independent of any browser.

We fire two server-side events. A login event after
the JWT is issued. And a session_analyzed event after
the Firestore write — including the overall score,
language, mode, source (local or Google Drive),
dominant emotion from Cloud Vision, and the number
of emotion frames detected.

This means even if a user closes their browser before
the result loads, we still have a full server-side
record that the analysis completed.

SLIDE 11 — Summary
─────────────────────────────────────────────────────────

That brings us to our final slide.

Let me start with the notable challenges we overcame.

The first was the Google Photos Library API deprecation.
We originally implemented the social media video picker
using the Google Photos Library API. During testing we
discovered Google deprecated that scope for projects
created after May 2024. We diagnosed it via token
introspection and API error logs, then switched to
the Google Drive API — which provides identical
functionality and is fully available for new projects.

The second was the Cloud Run 32 megabyte response limit.
When users tried to analyze large Google Drive videos —
we tested a 116 megabyte MOV file — the server could not
proxy that video to the client without hitting the hard
response size limit. We went through three approaches:
streaming the file, returning a download URL for the
client to fetch directly — which hit CORS restrictions
— and finally the solution we use now: the server
downloads from Drive directly, runs the entire Gemini
and Vision pipeline, and returns only the JSON session
ID. The large video never touches the browser.

The third was memory pressure on Cloud Run. Processing
a 116 megabyte video in memory — plus a Gemini file
upload, a GCS upload, ffmpeg frame extraction, and
Cloud Vision batch calls all running concurrently —
exceeded the default 1 gigabyte Cloud Run limit. We
increased the service to 2 gigabytes with 2 vCPUs,
and restructured the pipeline to run all operations
in parallel using Promise.all so the additional passes
add minimal wall-clock time.

The fourth was hand gesture detection across both
recording modes. For locally recorded videos,
MediaPipe runs client-side with no server cost.
For Drive videos, we added a third Gemini pass that
reuses the already-uploaded file URI — so gesture
analysis for Drive videos costs exactly one extra
API call with zero re-upload overhead.

Looking ahead, the most impactful next steps would be
a progress tracking chart showing score improvement
across sessions over time, real-time feedback during
recording using WebSockets, and a mobile-optimized
recording experience.

What we learned:

Chaining AI prompts dramatically improves quality.
Pass 2 is far more specific than any single-prompt
approach because it is grounded in Pass 1's findings.

Running analyses in parallel with Promise.all is
critical for user experience. Cloud Vision, frame
extraction, GCS upload, and Gemini all run concurrently
— so the total time is the longest individual step, not
the sum of all steps.

API deprecations are a real engineering concern. Correct
code, correct tokens, correct scopes — and still a broken
feature because the API itself changed. Knowing how to
read error logs, introspect tokens, and pivot to
alternatives is essential.

And server-side analytics via the GA4 Measurement
Protocol gives a far more accurate picture than
frontend tracking alone — especially for long async
operations like AI analysis where the user may
close their browser before results load.

Thank you to Professor Lynne Grewe for the guidance
throughout this project. We hope VocalIQ demonstrates
not just that we can build a full-stack application,
but that we thought carefully about architecture,
made real engineering decisions, and handled
real-world problems along the way.
