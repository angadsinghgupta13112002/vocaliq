Hello everyone. My name is Abhinay Konuri, and this
is our CS651 Web Systems Project 2 presentation for
Professor Lynne Grewe at Cal State East Bay.

With me today is my teammate Samuel Paul Chetty.

Our application is called VocalIQ — which stands for
Vocal Intelligence Quotient. VocalIQ is an AI-powered
speaking coach that gives you instant, personalized
feedback on how you speak.

The app is fully live and deployed at
vocaliq-956080638663.us-central1.run.app — and we will
be demonstrating it live during this video.

Here is a quick overview of the Google technologies we
use:

Google OAuth 2.0 for user sign-in — users log in with
their Google account and receive a JWT token.

Google Drive OAuth — a separate OAuth flow that lets
users browse and pull videos directly from their
Google Drive account into the coaching session.

Google Gemini 2.5 Flash — the AI engine that analyzes
your speech through a two-pass pipeline and generates
a full coaching report.

Google Cloud Vision API — extracts frames from your
video every 3 seconds and uses Face Detection to build
a second-by-second emotion timeline showing when you
looked nervous, confident, or anxious.

Google Cloud Firestore — stores all user data and
coaching session results.

Google Cloud Storage — stores the uploaded video and
audio files.

Google Cloud Run — where the entire application is
deployed as a containerized service.

Google Analytics GA4 — tracks user behavior on both
the frontend and backend.

And Google Cloud Logging — captures every server
event through Cloud Run's built-in log collection.

All of these are fully working and live right now.

SLIDE 3 — Deployment
─────────────────────────────────────────────────────────

Let me walk you through our deployment setup.

[OPEN: console.cloud.google.com → Cloud Run → vocaliq]

VocalIQ is deployed on Google Cloud Run under our
project auraboard-492122. Cloud Run is Google's fully
managed serverless container platform — it runs our
Docker container and automatically scales based on
traffic, including scaling to zero when idle.

Our build process uses a multi-stage Dockerfile.

Stage one uses a Node.js Alpine image to install
dependencies and run the Vite build command, which
compiles our React frontend into a static bundle.

Stage two copies that bundle alongside our Express
server code and runs Node.js. The Express server does
two things — it serves the React static files, and it
handles all API routes under slash api. Frontend and
backend are served from the exact same URL.

For security, we use Application Default Credentials.
There is no service account key file inside the Docker
image. The Cloud Run service has a service account
attached with exactly the permissions it needs —
Firestore, Cloud Storage, and Gemini.

[SHOW: Revisions tab in Cloud Run]

The Revisions tab shows our full deployment history.
You can see every time we pushed a new image and
deployed. Each revision has a timestamp and image
digest as an audit trail.

SLIDE 7 — Google Cloud Vision
─────────────────────────────────────────────────────────

Thank you Abhinay. I'm Samuel Paul Chetty, and I'll
walk you through the two AI systems powering VocalIQ —
Cloud Vision and Gemini — then cover Analytics,
Logging, and our project summary.

Google Cloud Vision is one of the two AI systems
running in VocalIQ. Specifically, we use Cloud Vision's
Face Detection feature to analyze the speaker's facial
expressions frame by frame.

Let me explain the full pipeline.

[OPEN: VS Code → client/src/utils/frameExtractor.js]

Step one happens entirely in the browser. When a user
submits a video, before it's uploaded to the server,
our frameExtractor.js utility uses the browser's
HTML5 Video element and Canvas API to extract one JPEG
frame every three seconds — up to twenty frames total.

Each frame is captured at 320 by 240 resolution and
encoded as a base64 JPEG string. The entire extraction
happens client-side — no server roundtrip, no ffmpeg,
no extra dependencies.

[OPEN: VS Code → server/services/visionService.js]

Step two is on the server. The frames array is sent
to our Express server alongside the video file as part
of the multipart form submission.

Our visionService.js takes those frames and sends
them to the Google Cloud Vision Face Detection API
in batches of five — to stay within rate limits —
using Promise.all for parallel processing.

For each frame, Cloud Vision returns likelihood scores
for four emotions — joy, sorrow, anger, and surprise —
rated on a scale from VERY_UNLIKELY to VERY_LIKELY.

We map these to coaching-specific labels. High joy
means confident. High sorrow means nervous. Combined
sorrow and low joy means anxious. High anger means
frustrated. We calculate a confidence score for each
detection and attach the timestamp.

[SHOW: the Emotion Timeline tab on the coaching report]

The result is the emotion timeline you saw in the demo
— each second mapped to an emotion, summarized into
dominant emotion, nervous moments, and confident moments.

Critically, the Vision analysis runs in parallel with
the Gemini pipeline on the server. Both start at the
same time. Since Cloud Vision processes twenty small
JPEG frames while Gemini processes a full video file,
Vision almost always finishes first — so it adds
nearly zero extra time to the total analysis duration.

SLIDE 8 — Google Gemini
─────────────────────────────────────────────────────────

The Gemini API is the core of VocalIQ. We use
Gemini 2.5 Flash through a two-pass chained pipeline.

Let me explain why we designed it this way.

A naive approach sends the video to Gemini with one
prompt asking for everything at once. The problem is
that Gemini produces generic, surface-level advice
when asked too many things simultaneously.

We wanted coaching that is grounded in what the
speaker actually did — specific to their recording,
not a template response.

[SHOW: the coaching report on screen]

In Pass 1, we upload the video file to the Gemini
File API — a dedicated endpoint for large media files.
We then send a structured prompt asking Gemini to
act as a speech assessment engine and return JSON.

That JSON contains the overall score from zero to ten,
five sub-scores across content clarity, emotional
delivery, vocal confidence, pacing, and audience
engagement, the emotion Gemini detected versus the
emotion needed for that scenario, the speaker's words
per minute, and a full timestamped transcript broken
into segments.

[POINT to Overview and Transcript tabs]

Everything on the Overview and Transcript tabs
came from Pass 1.

Pass 2 takes the entire Pass 1 JSON output and
injects it as context into a second prompt. We
tell Gemini — here is what you observed. Now act
as a personal speaking coach and build a detailed
improvement plan based specifically on these findings.

Because Pass 2 has Pass 1's analysis as its starting
point, the coaching is grounded in what was actually
observed. It returns detailed tip cards with titles,
explanations, exercises, and better phrasing examples.
It returns vocal control data with current versus target
WPM, tone feedback, breath control, and pause technique.
It returns strengths to keep and a seven-day practice plan.

[POINT to Tips and Vocal Control tabs]

The Tips and Vocal Control tabs are entirely Pass 2 output.

The key insight is that Pass 2 cannot hallucinate generic
advice — it is forced to respond to what Pass 1 actually
found. That's what makes the output genuinely useful.

SLIDE 9 — Google Analytics
─────────────────────────────────────────────────────────

VocalIQ uses Google Analytics 4 at two separate levels
— the frontend and the backend — and I want to explain
both because they work differently.

On the frontend, we load the gtag.js library in
index.html. We also built a custom analytics.js module
with trackPageView and trackEvent functions.

A PageViewTracker component inside React Router calls
trackPageView on every navigation — so every page in
the single-page app is tracked as a separate page view.

We fire custom events at every meaningful user action.
Login events when OAuth completes. session_submitted
when a user clicks Analyze. session_complete when
analysis finishes. And report_tab_viewed with the tab
name every time a user switches tabs on the report.

[OPEN: analytics.google.com → Reports → Realtime]

This is the GA4 Realtime dashboard. It shows active
users right now and which events are firing. If we
use the app while this is open, events appear here
in real time.

[OPEN: Reports → Engagement → Events]

The Events report shows every custom event we've
tracked, how many times it fired, and how many unique
users triggered it. session_submitted and
session_complete together tell us our completion rate.
report_tab_viewed tells us which parts of the report
users actually read.

Now for the backend analytics. We built an analytics
service on the Express server that uses the GA4
Measurement Protocol — a special HTTP API that lets
your server send events directly to Google Analytics
independent of any browser.

We fire two server-side events. A login event right
after the JWT is issued — confirming the OAuth flow
completed on the backend. And a session_analyzed event
after the Firestore write — including the overall score,
language, mode, dominant emotion from Cloud Vision,
and the number of emotion frames detected.

This means even if a user closes their browser before
the result loads, we still have a full server-side
record that the analysis completed.

SLIDE 11 — Summary
─────────────────────────────────────────────────────────

That brings us to our final slide.

Let me start with the notable challenges we overcame.

The first was the Google Photos Library API deprecation.
We originally implemented the social media video picker
using the Google Photos Library API with the
photoslibrary.readonly scope. During testing we
discovered that Google deprecated this scope for
projects created after May 2024. Our Cloud Run project
was new enough to be affected. We diagnosed this by
reading the API error logs, confirmed it via token
introspection, and switched to the Google Drive API —
which provides the same video browsing functionality
and is fully available for new projects.

The second challenge was integrating Cloud Vision
without adding latency to the user experience. The
Gemini pipeline already takes 30 to 90 seconds.
We solved this by running Cloud Vision in parallel
with Gemini using Promise.all — both pipelines
start simultaneously, so the total analysis time
does not increase.

The third was an Express 5 breaking change that
caused startup crashes until we replaced the
wildcard route syntax.

The fourth was Application Default Credentials —
our code tried to load a key file that doesn't
exist on Cloud Run, crashing the server until
we added the conditional fallback.

Looking ahead, the most impactful next steps for
VocalIQ would be a progress tracking chart showing
score improvement across sessions over time, real-time
feedback during recording using WebSockets, and a
mobile-optimized recording experience.

What we learned from building this project:

Chaining AI prompts dramatically improves quality.
Pass 2 is far more specific than any single-prompt
approach because it is grounded in Pass 1's findings.

API deprecations are a real-world engineering concern.
You can have correct code, correct tokens, and correct
scopes — and still have a feature fail because the API
itself was restricted. Knowing how to read error logs,
introspect tokens, and find alternative approaches is
an essential engineering skill.

Cloud Run with Application Default Credentials is the
correct pattern for server-side Google API access.
Never put service account key files in Docker images.

And the GA4 Measurement Protocol is a powerful tool
most developers skip — server-side event tracking
gives a far more accurate picture of user behavior
than frontend analytics alone.

Thank you to Professor Lynne Grewe for the guidance
throughout this project. We hope VocalIQ demonstrates
not just that we can build a full-stack application,
but that we thought carefully about architecture,
made real engineering decisions, and handled
real-world problems along the way.

We are Abhinay Konuri and Samuel Paul Chetty.

Thank you for watching.
