SLIDE 7 — Google Cloud Vision solution
─────────────────────────────────────────────────────────

This slide is about Google Cloud Vision. I want to be very
transparent here — VocalIQ does not use the standalone Cloud
Vision API, and I want to explain exactly why that was the
right decision for this application.

Google Cloud Vision is a powerful API. It can identify objects,
read text, detect faces, and analyze images. But it has an
important limitation — it only works on static images. It
cannot process video over time, it cannot hear audio, and it
cannot generate coaching advice.

For a speaking coach, you need all three of those things
simultaneously. You need to hear the speech, you need to see
the speaker's body language and delivery over time, and you
need to synthesize all of that into meaningful feedback.

Gemini 2.5 Flash is a natively multimodal model. When we
upload a video file to the Gemini File API, Gemini can see
every frame of that video AND hear the audio track, all in
a single API call. It understands context, generates natural
language analysis, and produces structured JSON output — none
of which Cloud Vision can do.

[OPEN: server/services/geminiCoachingService.js in VS Code]

You can see in our code here — we upload the video buffer to
the Gemini File API using the uploadFile method. The file URI
is then passed as a Part alongside our prompt in the
generateContent call. Gemini processes both the video frames
and audio together and returns a comprehensive analysis.

If we had used Cloud Vision instead, we would have needed
Cloud Vision for frame analysis, Cloud Speech-to-Text for
audio transcription, and a third separate system to synthesize
everything into coaching advice. That's three APIs, three
integrations, three billing items, and three potential failure
points — versus one Gemini call that does all of it better.

So our position is: Gemini 2.5 Flash makes Cloud Vision
redundant for this specific use case. It's not that we skipped
it — it's that we chose the better tool.

SLIDE 8 — Google Gemini solution
─────────────────────────────────────────────────────────

Thank you Abhinay. I'm Samuel Paul Chetty, and I'll be
explaining how we use the Google Gemini API in depth.

The Gemini API is the heart of VocalIQ. Without it, the app
is just a video recorder. Gemini is what turns a recording
into a coaching report.

We use Gemini 2.5 Flash through what we call a two-pass
chained pipeline. Let me explain what that means and why
we designed it this way.

[OPEN: the coaching report from the live demo — have it
visible on screen]

A naive approach would be to send the video to Gemini with
one prompt and ask for everything at once. The problem with
that is Gemini tends to give generic, surface-level advice
when asked too many things at once. We wanted specific,
grounded coaching — the kind of advice that references what
the speaker actually did, not a template response.

So we designed two separate passes that chain together.

In Pass 1, we upload the video or audio file directly to the
Gemini File API — this is different from just sending the
file in the prompt, it uses a dedicated upload endpoint that
handles large media files. We then send a structured prompt
that instructs Gemini to act as a speech assessment engine
and return a JSON object.

That JSON contains the overall score from zero to ten, five
sub-scores across content clarity, emotional delivery, vocal
confidence, pacing, and audience engagement, the emotion
Gemini detected in the speech versus what was needed for
that scenario, the speaker's words per minute, a full
timestamped transcript broken into segments, and the top
improvement areas.

[POINT to the Overview tab on the report]

Everything you see on the Overview tab — the score ring, the
sub-scores, the emotion box — that all came from Pass 1.

[POINT to the Transcript tab]

The Transcript tab is also Pass 1.

Now Pass 2 is where the deep coaching happens.

We take the entire JSON output from Pass 1 — all of it —
and we inject it into the Pass 2 prompt as context. We
essentially tell Gemini: "Here is what you found in this
speech. Now act as a personal speaking coach and build a
detailed improvement plan based specifically on these
findings."

Because Pass 2 already has Pass 1's analysis as context,
it produces coaching advice that is grounded in what was
actually observed. It's not generic. It references the
specific emotions detected, the specific WPM, the specific
weaknesses.

Pass 2 returns detailed tip cards — each one has a title,
an explanation, a specific practice exercise, and where
relevant, a better phrasing suggestion. It returns a vocal
control breakdown covering current pacing versus target
pacing, tone feedback, breath control guidance, and pause
technique. It returns a list of strengths to keep. And it
returns a full seven-day personalized practice plan.

[CLICK: Tips tab on the report]

This entire Tips tab is Pass 2 output.

[CLICK: Vocal Control tab]

And so is the Vocal Control tab — including these WPM
numbers right here.

The reason we chose a two-pass approach rather than a
single prompt is simple — Pass 2 cannot hallucinate generic
advice because it has Pass 1's real findings in front of
it. The coaching is always tied to what was actually in
the recording. That's what makes the output genuinely
useful rather than just impressive-looking.

Thank you Samuel. I'm Lasya Uma Sri Lingala, and I'll cover
how we implemented analytics and logging, and then wrap up
with our summary.

VocalIQ uses Google Analytics 4 at two completely separate
levels — the frontend and the backend — and I want to explain
both because they work differently and serve different purposes.

On the frontend, we load the gtag.js library in our index.html
file. This automatically initializes GA4 and tracks the
initial page load. We also built a custom utility module called
analytics.js with two functions — trackPageView and trackEvent.

We placed a component called PageViewTracker inside React
Router that calls trackPageView every time the route changes.
This means every navigation within the single-page app is
tracked as a page view — the dashboard, the session setup,
the recording page, the report — all of it shows up in GA4.

Beyond page views, we fire custom events at every meaningful
user action. When a user successfully logs in, we fire a
login event. When they click Analyze My Speech, we fire
session_submitted with the mode and scenario as parameters.
When the analysis comes back successfully, we fire
session_complete. And every time a user clicks a tab on
the coaching report, we fire report_tab_viewed with the tab
name — so we know whether users are engaging with the
Transcript, Tips, or Vocal Control sections.

[OPEN: analytics.google.com → Reports → Realtime]

You can see the GA4 Real-Time dashboard here. This shows
active users right now, which pages they're on, and which
events are firing. If we use the app while this is open,
you'll see the events appear here in real time.

[OPEN: Reports → Engagement → Events]

This is the Events report — it shows every custom event we've
tracked, how many times each one has fired, and how many
unique users triggered it. You can see session_submitted and
session_complete here, which together tell us our analysis
completion rate. You can see report_tab_viewed with counts
for each tab, which tells us what users actually read.

Now for the backend analytics — this is something most
applications don't do, but we felt it was important.

We built an analytics service on the Express server that uses
the GA4 Measurement Protocol. This is a special HTTP API that
lets your server send events directly to Google Analytics,
completely independent of any browser. We use axios to post
events to the Measurement Protocol endpoint with our GA4
API secret.

We fire two server-side events. The first is a login event
sent right after the JWT is issued on the server — this
confirms the OAuth flow completed successfully on the backend,
not just on the client. The second is a session_analyzed
event sent after the Firestore write — it includes the
overall score, the language detected, the mode, and the
scenario. This means even if a user closes their browser
before the result loads, we still have a record on the server
that the analysis completed.

Analytics failures are silently swallowed — if the GA4
Measurement Protocol call fails for any reason, it never
affects the API response the user gets.

SLIDE 11 — Summary
─────────────────────────────────────────────────────────

And that brings us to our final slide — the summary.

Let me start with the problems we hit, because I think these
are genuinely useful lessons.

The first was an Express 5 breaking change. Express 5 no longer
supports wildcard routes written as app.get with a star. It
throws a PathError at startup. We spent time debugging this
before realizing the fix was simply to replace it with
app.use and a handler function.

The second was a Cloud Run environment variable issue. Cloud
Run reserves the PORT variable and injects it automatically
into the container. When we tried to set PORT equals 8080 in
our deploy command's environment variables, the deployment
failed with an error saying reserved environment variable
names were provided. We removed it and let Cloud Run handle
it.

The third was an Application Default Credentials problem. Our
gitignore and dockerignore files correctly excluded the
service account key file from the Docker image — but our code
was trying to load that file unconditionally. On Cloud Run
where the file doesn't exist, the server crashed on startup.
The fix was to add a conditional — if the key file path
environment variable is set, use the key file, otherwise fall
back to Application Default Credentials.

The fourth was a Vite build-time environment variable issue.
Vite replaces environment variables at build time, not at
runtime. So the API URL of localhost colon 8080 was hardcoded
into the production bundle. We fixed it by creating a
.env.production file with VITE_API_URL set to slash api for
same-origin serving on Cloud Run.

Now for how we would improve VocalIQ going forward.

The most impactful improvement would be progress tracking —
showing users a trend of their scores across multiple sessions
so they can see whether they're actually improving over time.
Right now you can see your history but there's no visualization
of progress.

We would also want to add real-time feedback during recording
— not just after. Imagine a subtle indicator showing you're
speaking too fast while you're still recording, rather than
telling you afterward.

We would complete the Instagram OAuth integration from our
original proposal. And we would add multi-language support
for the coaching reports — Gemini already detects the language
of the speech, so it's a logical extension.

Finally — what we learned from building this.

The most important technical lesson was that chaining AI
prompts dramatically improves output quality. Pass 2 produces
coaching that is far more specific and useful than anything
we got from single-prompt approaches, precisely because it's
grounded in Pass 1's findings.

We learned that Application Default Credentials is the correct
way to handle secrets on Cloud Run — never put key files in
Docker images.

We learned that GA4's Measurement Protocol is a powerful tool
that most developers don't use — tracking server-side events
gives you a much more complete picture of user behavior than
frontend analytics alone.

And we learned that Vite, Gemini, Express 5, and Cloud Run
each have specific behaviors and constraints that are not
immediately obvious — and that the only way to learn them
is to build something real and hit the errors yourself.

Thank you to our professor for the guidance throughout this
project. We hope VocalIQ demonstrates not just that we can
build a full-stack application, but that we thought carefully
about the design decisions along the way.

Chetty, and Lasya Uma Sri Lingala — thank you for watching,
