Section 1: Google Analytics

1.0 Implementation Summary

VocalIQ implements Google Analytics 4 (GA4) at two distinct levels — client-side via the gtag.js library and server-side via the GA4 Measurement Protocol — providing a complete view of user behavior from both the browser and the backend.

GA4 Measurement ID: G-SN3LE9Y5RT

Client-Side (React.js): The gtag.js snippet is loaded in client/index.html, which initializes GA4 on every page load. A custom analytics.js utility module exposes two functions — trackPageView(path) and trackEvent(eventName, params) — which are called throughout the React application. A PageViewTracker component placed inside React Router fires a page view on every route change automatically.

// client/src/utils/analytics.js
const GA_ID = "G-SN3LE9Y5RT";

export const trackPageView = (path) => {
  if (typeof window.gtag !== "function") return;
  window.gtag("config", GA_ID, { page_path: path });
};

export const trackEvent = (eventName, eventParams = {}) => {
  if (typeof window.gtag !== "function") return;
  window.gtag("event", eventName, eventParams);
};

// client/src/App.jsx — PageViewTracker component
const PageViewTracker = () => {
  const location = useLocation();
  useEffect(() => {
    trackPageView(location.pathname + location.search);
  }, [location]);
  return null;
};

Server-Side (Node.js / Express): A dedicated analyticsService.js module on the Express backend sends events directly to GA4 using the Measurement Protocol HTTP API via axios. This ensures key server-side events — such as a successful login and completed speech analysis — are tracked independently of browser behavior, even if the user closes the tab mid-session.

// server/services/analyticsService.js
const axios = require("axios");

const GA_MEASUREMENT_ID = "G-SN3LE9Y5RT";
const GA_API_SECRET     = process.env.GA_API_SECRET || "";
const GA_MP_ENDPOINT    = `https://www.google-analytics.com/mp/collect
  ?measurement_id=${GA_MEASUREMENT_ID}&api_secret=${GA_API_SECRET}`;

const sendServerEvent = async (clientId, eventName, params = {}) => {
  if (!GA_API_SECRET) {
    console.log(`[analytics] ${eventName}`, { clientId, ...params });
    return;
  }
  try {
    await axios.post(GA_MP_ENDPOINT, {
      client_id: clientId || "server",
      events: [{
        name:   eventName,
        params: { ...params, engagement_time_msec: "1" },
      }],
    });
  } catch (err) {
    // Analytics failures never affect the main API response
    console.warn("[analytics] Measurement Protocol error:", err.message);
  }
};

module.exports = { sendServerEvent };

1.1.b — Interpretation of Trends

The page view data reveals which screens users spend the most time on within VocalIQ. The /report/:id page consistently recorded the highest average engagement time, indicating that users thoroughly review their coaching results — particularly the Tips and Vocal Control tabs. The /session/new setup page and the /session/record recording page also show high view counts, confirming that users are completing the full session flow rather than bouncing after login.

The trend in active users demonstrates organic growth from friend and colleague sharing, validating that the product's core value — instant personalized speaking feedback — resonates with new users. Returning user sessions (identified by persistent JWT tokens and repeat Firestore queries) indicate that several users completed more than one coaching session, suggesting meaningful engagement beyond a single novelty visit.

From an application health perspective, the ratio of page views on the analysis loading screen versus the report screen provides an implicit success rate for the Gemini pipeline — if users reach the report, the analysis completed successfully.

1.1.c — Limitations of Metric 1

The primary limitation of the active users and page views metric is that it does not distinguish between intentional navigation and accidental or exploratory clicks. A user who lands on the report page but immediately navigates away is counted the same as one who spends five minutes reading their coaching feedback. GA4's default session timeout of 30 minutes of inactivity also means that a user who pauses mid-recording and returns later may be counted as two separate sessions, inflating active user counts.

Additionally, because VocalIQ is a React single-page application, traditional page view counting requires explicit gtag("config", ...) calls on each route change. If any route transition is missed in the PageViewTracker implementation, those views would be undercounted. Finally, users who run ad-blockers or privacy-focused browsers may have the gtag.js script blocked entirely, creating a systematic undercount of privacy-conscious users.

1.2.b — Interpretation of Trends

The ratio of session_complete events to session_submitted events represents the analysis success rate of the Gemini two-pass pipeline. A high completion rate (above 90%) indicates that the Gemini File API is reliably processing video and audio uploads and that the JSON parsing of Gemini's response is stable.

Any gap between submission and completion events highlights failure cases — these could be caused by oversized files, unsupported MIME types, Gemini API rate limits, or network timeouts. By tracking this funnel in GA4, we can identify whether failures are isolated incidents or systematic issues requiring backend intervention.

The scenario parameter attached to both events also allows segmentation — for example, comparing completion rates for "Job Interview" sessions versus "Class Presentation" sessions. If one scenario type has significantly lower completion rates, it may indicate that certain speaking contexts produce longer videos that exceed processing limits.

The server-side session_analyzed event (sent via Measurement Protocol) provides a second confirmation signal from the backend, allowing cross-validation between client-reported completions and server-confirmed saves to Firestore.

1.2.c — Limitations of Metric 2

The primary limitation of the funnel metric is attribution ambiguity. If a user submits a session and the analysis completes on the server, but the user closes the browser tab before the response reaches the client, session_complete is never fired — even though the analysis succeeded. This creates an artificial drop in the funnel that does not reflect actual backend failures.

Conversely, if a network error causes the client to show an error toast but the server actually completed the analysis and saved to Firestore, the backend session_analyzed Measurement Protocol event would still fire, creating a discrepancy between client and server event counts.

Furthermore, the 30–90 second analysis window means users on slow connections who abandon the loading screen inflate the dropout rate without representing true failures. A more accurate metric would compare the server-side session_analyzed count against the number of Firestore documents actually written — a comparison that GA4 alone cannot provide without cross-referencing Firestore query counts.

1.3.b — Interpretation of Trends

The login event metric tracks user acquisition and retention simultaneously. Each new Google OAuth login that creates a new Firestore user document represents a new user acquisition — the app successfully converted a visitor into an authenticated user.

Comparing login event counts to unique user counts in Firestore reveals the return login rate — how many users came back to VocalIQ for a second or third session. A high return rate indicates that users found the coaching report valuable enough to want another session, which is the primary engagement goal of the application.

The dual tracking of the login event (both frontend and backend via Measurement Protocol) also allows us to validate GA4's client-side tracking accuracy. If the server-side login count consistently matches the client-side count, our analytics implementation is reliable. If there is a persistent gap, it suggests that some users' browsers are blocking the gtag.js script, quantifying the blind spot in our client-side analytics.

1.3.c — Limitations of Metric 3

The login metric is limited by Google OAuth's session persistence behavior. Because VocalIQ stores the JWT in localStorage and the React app reads it on page load, users who are already authenticated do not trigger the login event on subsequent visits — only their initial login is tracked per session. This means the login count underrepresents returning user engagement.

Additionally, the client_id sent in the Measurement Protocol login event is the Firebase UID, which is a stable identifier. However, GA4's own client ID (the _ga cookie) is a separate identifier. The two systems are not automatically linked, making it impossible to correlate server-side login events with the same user's frontend page view events in GA4 Explore without a custom user ID configuration.

Finally, the login event does not capture failed login attempts — users who clicked "Sign in with Google" but denied the OAuth consent, or whose OAuth exchange failed, are invisible to this metric. A more complete picture would require logging OAuth errors separately.

Section 2: Google Logging

2.0 Implementation Summary

VocalIQ implements Google Cloud Logging through two complementary mechanisms. On the backend, all console.log, console.warn, and console.error output from the Express server is automatically captured by Google Cloud Run's logging infrastructure and forwarded to Google Cloud Logging (formerly Stackdriver Logging) in real-time. On the frontend, structured event logging is handled through the GA4 analytics integration described in Section 1, with browser console output visible in browser DevTools.

All server-side log statements follow a consistent prefix format — [module] followed by the event description — making it straightforward to filter logs by module in the Cloud Logging console:

PrefixModuleWhat it logs
[coaching]coachingController.jsSession start, save confirmation, errors
[gemini]geminiCoachingService.jsFile upload, Pass 1 complete, Pass 2 complete, token counts
[storage]storageService.jsGCS upload URL, file size
[analytics]analyticsService.jsGA4 Measurement Protocol events (when no API secret)
[auth]authController.jsOAuth callback, JWT issued

2.1 Frontend Implementation (React.js)

The frontend logging implementation in VocalIQ is handled through two mechanisms: Google Analytics event tracking (which logs user interactions to the GA4 platform) and browser console logging for development-time diagnostics.

Analytics-Based Event Logging: Every significant user action is tracked as a GA4 event using the trackEvent utility. This creates a structured, queryable log of user behavior in the GA4 Events dashboard:

// client/src/pages/CoachingSession.jsx
import { trackEvent } from "../utils/analytics";

const handleSubmit = async () => {
  // Log the session submission event with context parameters
  trackEvent("session_submitted", {
    mode:     mode,
    scenario: context.scenario || "General"
  });

  try {
    const res = await analyzeSession(formData);

    if (res.data.success) {
      // Log successful analysis completion
      trackEvent("session_complete", {
        mode:     mode,
        scenario: context.scenario || "General"
      });
      navigate(`/report/${res.data.sessionId}`);
    }
  } catch (err) {
    toast.error(err.response?.data?.error || "Analysis failed.");
  }
};

// client/src/pages/CoachingReport.jsx — tab interaction logging
{["overview","transcript","tips","vocal"].map(([id, label]) => (
  <button
    key={id}
    className={`tab-btn${activeTab === id ? " active" : ""}`}
    onClick={() => {
      setActiveTab(id);
      // Log every tab switch as a named event with the tab identifier
      trackEvent("report_tab_viewed", { tab: id });

  >
    {label}

// client/src/pages/LoginPage.jsx — login event logging

  if (user) {

}, [user]);

2.2 Backend Implementation (Express / Node.js)

The backend logging in VocalIQ uses Google Cloud Run's built-in log aggregation, which automatically captures all standard output from the Node.js process and forwards it to Google Cloud Logging. Every significant server event — API calls to Gemini, GCS uploads, Firestore writes, and GA4 Measurement Protocol calls — is logged with a structured prefix for easy filtering.

Gemini API Call Logging: Every call to the Gemini File API is logged before and after the request, including the pass number, file size, and token usage on completion:

// server/services/geminiCoachingService.js
const analyzeVideoWithGemini = async (videoBuffer, mimeType, context) => {
  console.log("[gemini] Starting two-pass video analysis");

  // Upload file to Gemini File API
  console.log(`[gemini] Uploading video buffer — size: ${videoBuffer.length} bytes`);
  const fileResult = await fileManager.uploadFile(tmpPath, {
    mimeType,
    displayName: "vocaliq-analysis",
  });
  console.log(`[gemini] File uploaded — URI: ${fileResult.file.uri}`);

  // Pass 1 — Speech Assessment
  console.log("[gemini] Running Pass 1 — speech assessment");
  const pass1Response = await model.generateContent([pass1Prompt, videoPart]);
  console.log("[gemini] Pass 1 complete");

  // Pass 2 — Deep Coaching (using Pass 1 results as context)
  console.log("[gemini] Running Pass 2 — deep coaching plan");
  const pass2Response = await model.generateContent([pass2Prompt]);
  console.log("[gemini] Pass 2 complete");

  return { pass1: pass1Result, pass2: pass2Result };
};

// server/controllers/coachingController.js
const analyzeSession = async (req, res) => {
  try {
    const { uid } = req.user;
    console.log(`[coaching] Starting ${mode} analysis for user ${uid}`);

    // ... analysis runs ...

    await setDocument("coaching_sessions", sessionId, sessionData);
    console.log(`[coaching] Session saved: ${sessionId}`);

    // Log to GA4 via Measurement Protocol
    sendServerEvent(uid, "session_analyzed", {
      mode,
      scenario,
      language:      pass1.language     || "unknown",
      overall_score: pass1.overallScore || 0,
    });

  } catch (err) {
    console.error("[coaching] analyzeSession error:", err.message);
    res.status(500).json({ success: false, error: err.message });
  }
};
