PART 4 — Auth Controller (1:30 – 2:00)
Window: VS Code — authController.js
─────────────────────────────────────────────────────────

[OPEN: server/controllers/authController.js]

"Let me start with the authentication flow —
authController.js.

The googleLogin function is simple — it calls our OAuth
service to generate the Google authorization URL and
redirects the user to it.

The interesting part is googleCallback — this is the
function that runs after Google redirects back to our
server with the authorization code."

[SCROLL to the googleCallback function and highlight it]

"You can see here — we extract the code from the query
string, exchange it for user info via our OAuth service,
then upsert the user document in Firestore using set with
merge true — so existing user data is never overwritten,
only updated.

Then we issue a signed JWT token using the jsonwebtoken
library with our JWT_SECRET environment variable and a
seven-day expiry.

Notice this line — we call sendServerEvent here, sending
a login event to Google Analytics via the Measurement
Protocol. This is a server-side analytics event that fires
independently of the browser.

Finally we redirect back to the React app with the token
in the query string, where the frontend stores it in
localStorage."

PART 5 — Gemini Coaching Service (2:00 – 3:15)
Window: VS Code — geminiCoachingService.js
─────────────────────────────────────────────────────────

[OPEN: server/services/geminiCoachingService.js]

"This is the most important file in the entire project —
geminiCoachingService.js. This is where the two-pass
Gemini pipeline lives.

Let me scroll to the analyzeVideoWithGemini function."

[SCROLL to the main function]

"The function takes the video buffer, the MIME type, and
the context object as inputs.

The first thing we do is write the video buffer to a
temporary file on disk. This is necessary because the
Gemini File API requires a file path, not a buffer."

[HIGHLIGHT the file write section]

"Then we upload that file to the Gemini File API using
fileManager.uploadFile. This returns a file URI — a
reference to the uploaded file stored on Google's servers.
We log this URI — you'll see it appear in the Cloud Run
logs when a real analysis runs."

[SCROLL to the Pass 1 prompt section]

"Now this is Pass 1. We construct a prompt that instructs
Gemini to act as a speech assessment engine and return a
specific JSON structure. The prompt defines every field
we expect — the overall score, the five sub-scores, the
emotion detected, the emotion needed, the WPM, the
transcript array, and the improvement tips.

We call model.generateContent, passing in both the prompt
text and the video file as a Part. Gemini processes the
entire video — seeing the frames and hearing the audio —
and returns its assessment as text.

Because Gemini doesn't always return clean JSON — sometimes
it wraps it in markdown code fences — we have a JSON
extraction step here that strips any surrounding text and
parses just the JSON object."

[SCROLL to the Pass 2 section]

"Now Pass 2. Look at what we're doing here — we're taking
the entire Pass 1 JSON result and injecting it into the
Pass 2 prompt as a string. The prompt says, in effect —
here is what I found in this speech, now build a detailed
coaching plan based specifically on these findings.

This is the key design decision — Pass 2 is grounded in
Pass 1's actual observations. It cannot give generic advice
because it has real data in front of it. Every exercise,
every WPM target, every tip is tied to what was actually
found in the recording.

Pass 2 returns the detailed tip cards, vocal control
breakdown, strengths to keep, and practice plan — all
the content from the Tips and Vocal Control tabs you saw
in the app demo."

[OPEN: server/config/firebase.js]

"Let me quickly show firebase.js — specifically the

This is one of the more interesting infrastructure

[HIGHLIGHT the credential conditional block]

"You can see the conditional here. If the
GOOGLE_APPLICATION_CREDENTIALS environment variable is
set, we load the service account key file from that path —
this is for local development where we have the key file
on disk.

If that environment variable is NOT set — which is the
case on Cloud Run — we call admin.credential.applicationDefault().
Cloud Run automatically provides ADC credentials through
the service account attached to the Cloud Run service.

This means our Docker image contains no credentials
whatsoever. The key file is in gitignore and dockerignore.
Security is handled entirely by IAM roles assigned to the
Cloud Run service account."

PART 7 — Analytics Service (3:45 – 4:10)
Window: VS Code — analyticsService.js
─────────────────────────────────────────────────────────

[OPEN: server/services/analyticsService.js]

"This is our analytics service — the backend Google
Analytics implementation.

Most apps only track analytics on the frontend. We also
track from the server using the GA4 Measurement Protocol.

You can see the GA_MP_ENDPOINT — we post to Google Analytics
directly from Express using axios. The payload includes a
client_id which is the user's Firebase UID, and the event
name and parameters.

The critical part is this try-catch wrapper and the check
for GA_API_SECRET. If analytics fails for any reason —
network error, invalid secret, rate limit — we catch it,
log a warning, and move on. Analytics failures never
propagate to the main API response. The user experience
is never affected by an analytics error."

from the backend server via the Measurement Protocol
after the JWT is issued.

session_submitted fires when a user clicks Analyze
My Speech.

session_complete fires when the analysis returns
successfully — comparing this count to session_submitted
gives us our pipeline success rate.

report_tab_viewed fires every time a user clicks a tab
on the coaching report — so we know which sections users
actually engage with.

And session_analyzed fires from the backend server via
the Measurement Protocol — independent of the browser —
confirming the Firestore write completed successfully,
with the overall score and language as event parameters."

"Let me finish with the Dockerfile, which defines how
the application is packaged for Cloud Run.

We use a multi-stage build. Stage one uses Node.js Alpine
— a minimal Linux image — to install all client
dependencies and run the Vite build command. This compiles
the React application into a static bundle of HTML, CSS,
and JavaScript files.

Stage two starts fresh with another clean Node.js Alpine
image — this keeps the final image small by not including
all the build tools. It copies the compiled React bundle
from stage one, copies the server source code, installs
only the production server dependencies, and sets Node.js
as the entry point.

The result is a single Docker image that contains both the
React frontend as static files and the Express backend as
a Node.js process. When Cloud Run starts the container,
Express serves the static files for all regular page visits
and handles the API routes for slash-api requests — all
from the same port, the same origin, the same container."
